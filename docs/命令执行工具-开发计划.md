# 命令执行工具 — 开发计划

## 1. 背景分析

### 1.1 现状

NovaClaw **已经有一个功能完整的终端 WebSocket 系统**（`backend/src/server/ws/terminal.rs`），支持：

- 跨平台 shell 启动（Windows: PowerShell, Unix: bash）
- 实时 stdout/stderr 流式传输
- Ctrl+C 信号处理（跨平台）
- 工作目录管理
- 会话创建/销毁/重启
- Windows GBK 编码自动转 UTF-8

**缺的**：LLM Agent **无法调用**这个终端。Agent 没有 `execute_command` 工具，无法在 ReAct 循环中执行 shell 命令（编译、运行测试、安装依赖等）。

### 1.2 参考设计

参考三个项目：

| 特性 | Claude Code Rust | OpenCode | picoclaw | NovaClaw 已有 |
|---|---|---|---|---|
| 终端类型 | 普通子进程 | **PTY** 伪终端 | **PTY + pipe** 双模式 | 交互式 shell |
| 流式输出 | ❌ 一次性 | ✅ 实时 | ✅ 实时 | ✅ WS 已有 |
| 超时处理 | ✅ `tokio::time::timeout` | ✅ AbortSignal | ✅ `context.WithTimeout` | ❌ 需新增 |
| 杀进程组 | ❌ 不杀 | ✅ kill | ✅ **进程组 SIGKILL** | 仅 `child.kill()` |
| 输出截断 | ❌ 无 | ✅ 文件存储 | ✅ **10KB 同步 / 1MB 后台** | ❌ 需新增 |
| 危险命令拦截 | ❌ 无 | ✅ 分级权限 | ✅ **~40 条正则黑名单** | ❌ 需新增 |
| 进程隔离 | ❌ 无 | ❌ 无 | ✅ **Win 受限令牌 + Linux bwrap** | ❌ 无 |
| 执行模式 | 仅同步 | 同步 + PTY | ✅ **同步 + 后台双模式** | 仅交互式 WS |
| 路径穿越防护 | ❌ 无 | ❌ 无 | ✅ **重新解析符号链接防 TOCTOU** | ❌ 无 |
| 取消机制 | ❌ 无 | ✅ AbortSignal | ✅ `terminateProcessTree` | ✅ AtomicBool |

### 1.3 关键设计决策

NovaClaw 的终端系统是 **交互式 shell**（用户通过 WS 实时交互），而 LLM 需要的命令执行是 **"一次性"运行并返回结果**。这两个场景不同，但可以复用同一个基础设施。

## 2. 架构方案

### 方案 A（当前推荐）：基于 portable-pty

```

LLM 调用 execute_command 工具
  → 后端通过 portable-pty 创建伪终端
    → 在 PTY 中执行命令
      → 实时读取 PTY 输出（字节流）
        → 超时后 terminate_process_tree()
          → 收集所有输出 + exit code → 返回给 LLM
```

### 方案 B：复用现有 WS 终端（pipe 方式，已否决）

直接复用现有 `ws/terminal.rs` 的 `TerminalManager`，通过 stdin/stdout pipe 执行命令。

**否决原因**：pipe 方式下命令检测到 stdout 不是 TTY 时会禁用颜色输出、缓冲输出、甚至改变行为（如 `npm install` 不显示进度条）。PTY 解决这些问题，且 `portable-pty` 跨平台支持良好。

### 推荐：方案 A（PTY）

技术栈变更：

```
之前方案：复用 TERMINAL_MANAGER.create_session() ← pipe 方式
当前方案：portable-pty::new() 创建 PTY ← 伪终端方式
```

使用 PTY 后，命令行为与用户在真实终端执行完全一致。

注意：`ws/terminal.rs` 的 pipe 终端保留给前端交互式 WebSocket 终端使用，两者共存不冲突。

### portable-pty 集成方式

`portable-pty` 由 WezTerm 团队维护，跨平台（Windows/macOS/Linux）：

```rust
use portable_pty::{PtySize, native_pty_system, CommandBuilder};

let pty_system = native_pty_system();
let pair = pty_system.openpty(PtySize {
    rows: 24, cols: 80,
    pixel_width: 0, pixel_height: 0,
}).unwrap();

let mut child = pair.slave.spawn_command(
    CommandBuilder::new("powershell")
        .cwd(workdir)
        .arg("-NoProfile")
        .arg("-Command")
        .arg(command)
).unwrap();

// 从 PTY master 读取输出（可跨平台流式读取）
let mut reader = pair.master.try_clone_reader().unwrap();
// 读取直到 EOF 或超时
```

PTY 模式下，`stdout` 和 `stderr` 合并为一个输出流（终端行为），输出包含 ANSI 转义序列。

### 输出处理

PTY 输出包含 **ANSI 转义序列**（颜色、光标控制等），需要处理：

```rust
// 选项 1：剥离 ANSI 序列（推荐，简化 LLM 处理）

// 选项 2：保留原始输出（更真实，但 LLM 需要处理转义码）
// 返回原始字节，由前端渲染
```

推荐**选项 1**：使用 `strip-ansi-escapes` crate 清洗输出后再返回给 LLM。

### Phase 2：跨平台 shell 逻辑（~0.5h）

```rust
fn build_command(command: &str, workdir: &Path) -> (String, Vec<String>) {
    if cfg!(target_os = "windows") {
        let shell = "powershell.exe";
        let args = vec!["-NoProfile", "-NonInteractive", "-Command", command];
        (shell.to_string(), args.iter().map(|s| s.to_string()).collect())
    } else {
        let shell = "bash";
        let args = vec!["-c", command];
        (shell.to_string(), args.iter().map(|s| s.to_string()).collect())
    }
}
```

## 6. 与已有系统的关系

```
                    SSE 流
execute_command ←─────────→ 聊天框（前端）
      │
      │ 调用
      ▼
TERMINAL_MANAGER.create_session()
      │
      ├─ tokio::process::Command::new("powershell" / "bash")
      │     .current_dir(cwd)
      │     .stdin(Stdio::piped())
      │     .stdout(Stdio::piped())
      │     .stderr(Stdio::piped())
      │
      ├─ session.write_stdin(command + "\r\n")
      │
      ├─ loop 读取 stdout/stderr（带 timeout）
      │
      └─ 超时 → session.kill() → 收集输出 → 返回
```

**不冲突**：WebSocket 终端（用户手动交互）和 `execute_command` 工具（LLM 调用）共用同一个 `TerminalManager`，但会话隔离，互不影响。
