# DeepSeek 缓存命中率核心原理

## 一、高缓存命中率的根本因素

DeepSeek API 的缓存机制本质上很简单：

> **同一个请求中开头的字节序列，如果和之前某个请求完全一致（精确到每个 bit），这些 Token 就算作「缓存命中」，按低价计费。**

所以根本因素只有一个：

### 每次请求的「前 N 个字节」必须一模一样

不是"大概一样"、不是"语义相同"、而是**二进制级别完全一致**。一个空格的增减、一个 JSON key 的顺序变化、一个 None 字段写成了 null，都会让缓存 miss。

---

## 二、冻结前缀到底冻结了什么？

很多人以为只冻结了 system prompt，但其实**冻结的是请求头部的一整块区域**。

### 一次请求的完整结构图

```
┌────────────────────────────────────────────────────┐
│ 请求头部（冻结前缀 — 整个会话期内固定不变）           │
│                                                      │
│  system: {                   ← frozen_system_prompt  │
│    "你是一个AI助手..."                                │
│    "## System Rules"                                  │
│    "## Output Format"                                 │
│    "Language Requirement"                             │
│    "Tool Call Termination Rules"                      │
│    ...                                                │
│    "---"     ← 冻结边界                               │
│  }                                                    │
│                                                      │
│  // ↑↑↑ 以上所有字节，第二轮起完全相同 ↑↑↑             │
│  //     缓存命中候选，不计费或低计费                    │
├────────────────────────────────────────────────────┤
│ 追加日志（AppendOnlyLog — 单调增长）                  │
│                                                      │
│  messages: [                                         │
│    { user: "帮我分析项目" },                           │
│    { assistant: "好的..." tool_calls: [...] },        │
│    { tool: "结果..." },                               │
│    { user: "继续优化" },                              │
│    { assistant: "好的..." tool_calls: [...] },        │
│    ...                                                │
│  ]                                                    │
│                                                      │
│  // ↑↑↑ 前缀部分（除最后几轮）也是稳定的缓存命中 ↑↑↑   │
│  //     只有新增的末尾几条是新内容                      │
├────────────────────────────────────────────────────┤
│  volatile 后缀（每次变化，追加到最后一个 user 消息）    │
│                                                      │
│  "## Persistent Memory"                              │
│  "2026-05-28 14:30:00"                              │
│  "## Available Skills"                               │
│  ...                                                  │
└────────────────────────────────────────────────────┘
```

### 具体冻结的内容

#### ① frozen_system_prompt — 系统提示词

在会话期内、整个循环中完全不变的部分：

| 层 | 内容 | 是否冻结 | 原因 |
|:--:|------|:-------:|------|
| 1 | SOUL.md 身份层（AI 人格定义） | ✅ **冻结** | 会话期内不变 |
| 2 | Persistent Memory（跨会话记忆） | ❌ **不冻结** | 随用户操作变化，放入 volatile |
| 3 | System Rules（系统规则） | ✅ **冻结** | 固定不变 |
| 4 | Output Format（输出格式） | ✅ **冻结** | 固定不变 |
| 5 | --- 分隔线 | ✅ **冻结** | 固定边界 |
| 6 | Environment（日期、时间等环境信息） | ❌ **不冻结** | 每次变化，放入 volatile |
| 7 | Skills（技能索引列表） | ❌ **不冻结** | 可能变化，放入 volatile |

#### ② tools — 工具 Schema

工具 schema 在请求体中的位置同样属于**前缀区域**。工具 schema 的序列化也必须保证稳定性：

- 工具必须按名称排序输出，确保每次顺序一致
- 工具定义（名称、描述、参数）在会话期内不能随意变化
- 新增/删除工具会导致前缀变化，触发一次缓存未命中

## 三、三层上下文分区架构

缓存命中的架构基础来自三层分区：

```
┌─────────────────────────────────────────┐
│ IMMUTABLE PREFIX                        │ ← 会话期间固定
│   system + tool_specs + few_shots        │   缓存命中候选
├─────────────────────────────────────────┤
│ APPEND-ONLY LOG                         │ ← 单调增长
│   [assistant₁][tool₁][assistant₂]...    │   保持先前回合的前缀
├─────────────────────────────────────────┤
│ VOLATILE SCRATCH                        │ ← 每回合重置
│   思考过程, 临时规划状态                │   永不发送给 API
└─────────────────────────────────────────┘
```

**三个不变约束：**
1. **Prefix 会话内只计算一次**——hash 固定，后续请求不会因拼写差异导致缓存未命中
2. **日志只追加（Append-Only）**——不允许原位重写，确保历史字节序列稳定
3. **Scratch 区域永远不进入 API 请求**——思考内容不污染前缀

## 四、四重保缓存命中机制

### 机制一：前缀冻结（frozen_system_prompt）

- System prompt、tool specs、few-shot 示例在会话开始时冻结
- 每次请求发送完全相同的前缀字节序列
- 通过指纹（fingerprint）检测前缀是否发生变化

### 机制二：只追加日志（AppendOnlyLog）

- 历史消息只追加在末尾，不重排、不修改
- 原位修改历史消息会破坏后续所有消息的缓存
- 追加方式几乎不损失缓存命中率

### 机制三：易失性暂存区（Volatile Scratch）

- 模型的思考过程（reasoning_content）、临时规划状态等视为"易失性内容"
- 这些内容**永不发送给 API**，只用于本地展示和决策
- 避免了思考内容进入上下文破坏前缀稳定性

### 机制四：序列化字节锁定

- `LogEntry` 序列化时，None 字段跳过（`skip_serializing_if = "Option::is_none"`）
- 字段按字母序排列，确保 JSON 输出稳定
- 不发送 `reasoning_content` 到 API（`reasoning_content: None`）

## 五、为什么能做到 90%+ 缓存命中率

假设 10 轮对话，每轮输入 5000 tokens：

| 轮次 | 请求内容 | 缓存命中情况 | 命中率 |
|:----:|---------|:-----------:|:-----:|
| 第 1 轮 | `[前缀 + 第1轮对话]` | 全部未命中 | 0% |
| 第 2 轮 | `[前缀 + 第1轮 + 第2轮]` | 前缀+第1轮命中 | 50% |
| 第 3 轮 | `[前缀 + 第1-2轮 + 第3轮]` | 前缀+第1-2轮命中 | 66% |
| 第 4 轮 | `[前缀 + 第1-3轮 + 第4轮]` | 前缀+第1-3轮命中 | 75% |
| 第 5 轮 | `[前缀 + 第1-4轮 + 第5轮]` | 前缀+第1-4轮命中 | 80% |
| ... | ... | ... | ... |
| 第 10 轮 | `[前缀 + 第1-9轮 + 第10轮]` | 前缀+第1-9轮命中 | **90%** |

**随对话轮次增加，缓存命中率自然趋近 100%**，因为前缀中的"固定部分"越来越大，"新增部分"每次只有一小块。

## 六、最直观的理解方式

把每次给 DeepSeek 发送的请求想象成一本书：

| 轮次 | 书的内容 | 缓存命中？ |
|:----:|---------|:--------:|
| **第 1 轮** | `[前言 + 第1章 + 第2章]` | ❌ 新书，无缓存 |
| **第 2 轮** | `[前言 + 第1章 + 第2章 + 第3章]` | ✅ 前言+第1-2章命中 |
| **第 3 轮** | `[前言 + 第1章 + 第2章 + 第3章 + 第4章]` | ✅ 前言+第1-3章命中 |
| ❌ 别的框架 | `[前言 + 第3章(重排) + 第2章(重排) + 第4章]` | ❌ 全部 miss |

**冻结前缀 = 保证前言（system prompt）和前面的章节（历史对话）每次完全一样，只有最后新加的一章是新内容。**

## 七、影响缓存命中率的常见陷阱

### ❌ 破坏缓存的行为
- 在 messages 中插入时间戳、日期等动态内容
- 每次请求重新序列化 tool schema（顺序不固定）
- 修改旧消息的内容（如编辑 tool 返回结果）
- 将思考过程（reasoning_content）写入 API 请求
- 使用不同的空格/换行风格序列化 JSON

### ✅ 保持缓存的行为
- 前缀会话期内冻结，只构建一次
- 消息只追加，不修改已有内容
- 序列化设置确定性规则（字段排序、None 跳过）
- 动态内容（日期、记忆等）放入 volatile 后缀
- 工具 schema 按名称排序

## 八、两套压缩机制的适配场景

| 维度 | 旧机制（条数触发） | 新机制（Token % 触发） |
|:----:|:---------------:|:-------------------:|
| **触发条件** | 消息数 > 阈值（默认 40 轮） | Token 使用率 > 窗口的 85% |
| **适配模型** | 小上下文窗口模型（8K/16K/32K） | 超大上下文窗口模型（1M，如 DeepSeek） |
| **与窗口关系** | 与窗口大小无关 | 与窗口大小强相关 |
| **典型场景** | 本地部署 LLM | DeepSeek V4 系列 |

两套机制互不干扰，形成完美互补：

| 场景 | 谁触发？ |
|:----|:-------:|
| 小模型 32K 窗口，40 轮对话 | 旧机制（消息数达到阈值） |
| DeepSeek 1M 窗口，500 轮对话 | 新机制（Token 使用率达到 85%） |
| 中型窗口 128K，300 轮对话 | 两者都可，先到先触发 |

## 九、总结

```
缓存命中率 = f(前缀字节稳定性, 消息格式对齐, 序列化一致性)
              ↓               ↓              ↓
         完全对齐 ✅        完全对齐 ✅      完全对齐 ✅
```

| 决定性因素 | 保证措施 |
|:----------|:--------|
| **前缀字节序列稳定性** | `frozen_system_prompt` + `AppendOnlyLog` + `volatile_suffix` + 指纹检测 |
| **消息格式对齐** | DeepSeek V4 聊天模板 + DSML 标记 + 思考模式适配 |
| **序列化一致性** | `LogEntry` 确定性序列化 + None 字段跳过 + 工具 schema 排序 |
