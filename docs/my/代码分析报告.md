### C1. 命令执行工具 - 黑名单绕过漏洞

**文件**: [execute.rs](file:///d:/Project/novaclaw/backend/src/tools/execute.rs#L53-L96)  
**相关代码**:
```rust
fn check_command_deny<'a>(command: &str, patterns: &'a [String]) -> Option<&'a str> {
    // ...
    let cmd_lower = command.to_lowercase();
    for pattern in patterns {
        let pat_lower = pattern.to_lowercase();
        // 对多词模式（含空格）使用精确子串匹配
        if pat_lower.contains(' ') {
            if cmd_lower.contains(&pat_lower) {
                return Some(pattern);
            }
        } else {
            // 对单词模式使用词边界匹配
            // ...
        }
    }
}
```

**问题描述**:  
命令黑名单检查存在多个绕过方式：
1. **Base64/编码绕过**: 攻击者可以使用 `echo cm0gLXJmIC8= | base64 -d | bash` 绕过 `rm -rf` 检测
2. **变量替换绕过**: 使用 `R="rm"; $R -rf /` 可绕过检测
3. **多词模式匹配不完整**: `rm -rf` 的匹配是子串匹配，但 `rm  -rf` (多个空格) 会绕过

**影响**: 恶意 Prompt 注入可执行破坏性命令，导致文件系统损坏或数据丢失。

**建议**: 采用"分级权限 + 混合策略"方案（推荐）

黑名单的本质是"已知坏的，阻止它"，存在根本性缺陷——永远无法穷举所有危险命令。白名单"已知好的，允许它"安全模型更优，但对 AI Agent 可能过于严格。推荐以下混合方案：

```rust
// 1. 安全命令（直接执行，无需确认）
const SAFE_COMMANDS: &[&str] = &[
    "ls", "dir", "cat", "type", "grep", "find",
    "git status", "git diff", "git log",
    "cargo check", "cargo test", "npm test",
];

// 2. 需要用户确认的命令（Agent 执行前请求批准）
const NEEDS_APPROVAL: &[&str] = &[
    "cargo build", "cargo run", "npm run build",
    "python", "node", "cargo clippy",
];

// 3. 危险命令（一律阻止，不可绕过）
const BLOCKED_COMMANDS: &[&str] = &[
    "rm -rf", "format", "shutdown", "curl | bash",
    "del /f", "Remove-Item", "diskpart",
];

// 4. 未知命令 → 默认需要用户确认（安全兜底）
```

**额外加固措施**:

1. **命令规范化**（在安全检查前执行）
   ```rust
   fn normalize_command(cmd: &str) -> String {
       // 去除多余空格、统一大小写、解析简单变量
       // 然后再做安全检查
   }
   ```

2. **沙箱执行**
   - 使用容器（Docker）或沙箱运行命令
   - 限制文件系统访问范围（仅 workspace）
   - 限制网络访问（可选）

3. **语义分析**
   - 不仅检查命令字符串，还分析命令意图
   - 例如：`cat file.txt` 安全，但 `cat /etc/shadow` 危险

**方案对比**:

| 方案 | 安全性 | 可用性 | 适用场景 |
|------|--------|--------|----------|
| 纯黑名单（当前） | ⭐⭐ | ⭐⭐⭐⭐ | 信任用户的环境 |
| 纯白名单 | ⭐⭐⭐⭐⭐ | ⭐⭐ | 高安全要求环境 |
| **分级混合模式** | ⭐⭐⭐⭐ | ⭐⭐⭐⭐ | **推荐方案** |

### C2. 文件操作API - 路径遍历漏洞

**文件**: [files.rs](file:///d:/Project/novaclaw/backend/src/server/routes/files.rs#L88-L106)  
**相关代码**:
```rust
async fn read_file(Json(req): Json<ReadReq>) -> Json<serde_json::Value> {
    match std::fs::read_to_string(&req.path) {
        Ok(content) => Json(serde_json::json!({ "success": true, "data": content })),
        Err(e) => Json(serde_json::json!({ "success": false, "message": format!("读取文件失败: {}", e) })),
    }
}

async fn write_file(Json(req): Json<WriteReq>) -> Json<serde_json::Value> {
    // ...
    match std::fs::write(&req.path, &req.content) {
        Ok(_) => Json(serde_json::json!({ "success": true })),
        // ...
    }
}
```

**问题描述**:  
文件读写API直接使用用户传入的路径，未进行任何路径验证或沙箱限制。攻击者可以：
- 读取系统敏感文件：`{"path": "/etc/passwd"}` 或 `{"path": "C:\\Windows\\System32\\config\\SAM"}`
- 写入任意位置：`{"path": "../../.ssh/authorized_keys", "content": "..."}`
- 覆盖应用配置文件

**影响**: 任意文件读写可导致敏感信息泄露、系统配置被篡改、甚至远程代码执行。

**建议**:
- 实现路径沙箱：所有文件操作限制在 workspace 目录内
- 使用 `path.canonicalize()` 解析真实路径后验证是否在允许范围内
- 禁止访问敏感目录（系统目录、配置文件等）

### C4. 内存存储 - 并发竞态条件

**文件**: [store.rs](file:///d:/Project/novaclaw/backend/src/memory/store.rs#L49-L62)  
**相关代码**:
```rust
fn write_entries(&self, entries: &[String]) -> Result<(), String> {
    let inner = self.inner.lock().unwrap();
    let content = entries.join("\n\n§ ");
    let tmp = inner.memory_path.with_extension("md.tmp");
    fs::write(&tmp, &content).map_err(|e| format!("写入失败: {}", e))?;
    fs::rename(&tmp, &inner.memory_path).map_err(|e| format!("保存失败: {}", e))?;
    Ok(())
}

pub fn add_memory(&self, content: &str, _category: &str) -> Result<(), AppError> {
    // ...
    let inner = self.inner.lock().unwrap();
    let mut file = fs::OpenOptions::new()
        .create(true)
        .append(true)
        .open(&inner.memory_path)?;
    let existing_text = fs::read_to_string(&inner.memory_path).unwrap_or_default();
    // ...
}
```

**问题描述**:  
`add_memory` 方法存在TOCTOU (Time-of-Check-Time-of-Use) 竞态条件：
1. 获取锁后打开文件进行追加
2. 但在打开文件后，又读取文件内容检查重复
3. 如果两个线程同时调用 `add_memory`，可能导致：
   - 重复记忆被写入
   - 文件内容损坏（并发追加）

**影响**: 并发场景下记忆数据可能不一致或损坏。

**建议**:
- 在单个锁保护下完成"读取-检查-写入"整个操作
- 使用原子操作或事务机制
- 考虑使用SQLite等数据库替代文件存储

### H5. 前端SSE解析 - 内存泄漏风险

**文件**: [useApi.ts](file:///d:/Project/novaclaw/src/hooks/useApi.ts#L52-L130)  
**相关代码**:

### H6. 配置热更新 - 竞态条件

**文件**: [execute.rs](file:///d:/Project/novaclaw/backend/src/tools/execute.rs#L13-L32)  
**相关代码**:
```rust
pub static DENY_PATTERNS: once_cell::sync::Lazy<RwLock<Vec<String>>> =
    once_cell::sync::Lazy::new(|| RwLock::new(Vec::new()));

pub fn load_deny_patterns() -> Vec<String> {
    let patterns = match crate::APP_STATE.try_read() {
        Ok(state) => state.config.deny_patterns.clone(),
        Err(_) => {
            tracing::warn!("[ExecTool] APP_STATE try_read 失败（锁竞争）");
            if let Ok(cached) = DENY_PATTERNS.read() {
                if !cached.is_empty() {
                    return cached.clone();

            }
            Vec::new()  // ⚠️ 锁竞争时返回空列表！
        }
    };
    // ...
```

**问题描述**:

- 命令黑名单暂时失效
- 危险命令可能被执行

**影响**: 高并发时安全策略可能被绕过。

**建议**:
- 使用 `read().await` 替代 `try_read()`，确保总能获取最新配置
- 或在fallback时使用缓存的旧值而非空列表
- 考虑使用无锁数据结构（如 `arc-swap`）

**相关代码**:
```rust
fn save_image_data_url(data_url: &str, session_id: &str) -> Result<String, String> {
    let after_comma = data_url.find(',').ok_or("Invalid data URL format")?;
    let b64 = &data_url[after_comma + 1..];
    
    let data = base64::engine::general_purpose::STANDARD
        .decode(b64)
        .map_err(|e| format!("Base64 decode failed: {}", e))?;
    
    let filename = format!("{}.{}", uuid::Uuid::new_v4(), ext);
    let filepath = dir.join(&filename);
    std::fs::write(&filepath, &data)?;
    Ok(filename)
}
```

**问题描述**:  
1. 未限制上传图片的大小（可能接收100MB+的图片）
2. 未验证图片内容（可能上传恶意文件）
3. 图片永久存储，无清理机制

**影响**: 磁盘空间耗尽风险，可能的恶意文件上传。

**建议**:
- 限制图片最大大小（如10MB）
- 验证文件魔数（magic number）确保是真实图片
- 实现图片定期清理策略
