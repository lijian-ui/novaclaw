# NovaClaw 记忆系统设计方案

> 基于 Hermes Agent / OpenClaw / OpenCode 三个开源项目的记忆系统分析

---

## 0. 核心区分：记忆 ≠ 消息记录

这是最重要的概念。三个项目都把这两者严格分开：

| | 消息记录 (Transcript) | 记忆 (Memory) |
|---|---|---|
| **是什么** | 对话的完整日志，"谁说了什么" | 从中提炼的持久化知识 |
| **存什么** | 原始消息流 (role + content + tool_calls) | 事实、偏好、决策、环境知识 |
| **格式** | JSONL / SQLite 结构化表 | Markdown 文件 / SQLite 索引入口 |
| **检索** | 按时间顺序读取 | 关键词 / 语义搜索 |
| **例子** | `{"role":"user","content":"我喜欢简洁的回复"}` | `"User prefers concise responses"` |

## 1. 三个项目对比

### 1.1 总览

| | Hermes Agent | OpenClaw | OpenCode |
|---|---|---|---|
| **消息存储** | SQLite | JSONL | SQLite (Event Sourcing) |
| **记忆存储** | MEMORY.md + 可选外部 Provider | MEMORY.md + SQLite 索引 | Compaction 摘要 + AGENTS.md |
| **有无向量检索** | 有 (Holographic 等) | 有 (sqlite-vec) | 无 |
| **检索方式** | 关键词 + 语义 | 混合 (FTS5 + Vector) | 无检索，摘要替代旧消息 |
| **核心哲学** | Agent 主动管理记忆 | 文件系统 + 自动索引 | "记忆就是对话的聪明压缩" |

### 1.2 详细对比

#### Hermes Agent — 双层记忆架构

```
Layer 1: 内置 Memory (MEMORY.md)
  - § 分隔的声明式事实条目
  - Agent 调用 memory 工具添加/删除
  - Session 启动时加载为 frozen snapshot → 注入 system prompt
  - 约 800 tokens 容量限制

Layer 2: 外部 Provider (可选)
  - Honcho / Holographic / mem0 等
  - 每轮 prefetch 用户消息 → 语义搜索 → 注入 user message
  - Session 结束 / Context 压缩时同步
  - 支持 FTS5 + Vector 混合检索

时间维度:
  发送前 → prefetch 检索记忆 → 注入上下文
  运行中 → Agent 调用 memory 工具写入
  每 N 轮 → Nudge 触发后台 review
  结束时 → on_session_end() 提取最终记忆
```

#### OpenClaw — 文件 + 索引架构

```
Source:   MEMORY.md / memory/YYYY-MM-DD.md (文件系统)
           ↓ (Chokidar 文件监听 + 增量检测)
Index:    SQLite (FTS5 全文搜索 + sqlite-vec 向量搜索)
           ↓ (Hybrid Search)
Usage:    System Prompt 注入 (当前记忆)
          Dreaming Sweep (离线评分 + 短期记忆提升为长期)

触发时机:
  - 文件变更 → debounce 1.5s → 自动索引
  - Session 开始 → 同步索引
  - 搜索请求 → 按需强制同步
  - 周期性 → intervalMinutes 配置
```

#### OpenCode — 无独立记忆，Compaction 即记忆

```
无 MEMORY.md，无向量检索

机制 1: Compaction (上下文压缩)
  触发: 当 tokens >= model.limit.input - 20000
  动作: 调用 "compaction" agent 生成结构化摘要
        Goal / Progress / Decisions / Next Steps / Critical Context
  注入: 摘要作为普通 assistant 消息参与后续对话

机制 2: Instruction (项目指令)
  文件: AGENTS.md / CLAUDE.md
  注入: 每次 LLM 调用都作为 system prompt

机制 3: 模型自主记忆
  Prompt 引导模型自行维护 .github/instructions/memory.instruction.md
```

## 2. 记忆系统的 5 维度拆解

### 2.1 怎么存

| 项目 | 存储方式 | Schema |
|------|---------|--------|
| Hermes | MEMORY.md (纯文本, `§` 分隔) + SQLite (外部 Provider) | 自由文本条目，无结构 schema |
| OpenClaw | MEMORY.md → SQLite 索引 chunks 表 | `chunks(id, path, text, embedding, start_line, end_line)` |
| OpenCode | SQLite message/part 表 (compaction 产出的 assistant message) | 结构化 Markdown (Goal/Progress/Decisions/Next Steps) |

**共同点**：都认为纯文本文件（MEMORY.md）是记忆的"源"，数据库是"索引"。

### 2.2 什么时候存

| 触发条件 | Hermes | OpenClaw | OpenCode |
|---------|--------|----------|----------|
| Agent 主动调用工具 | ✅ `memory add` | ❌ | ❌ (模型自主) |
| 周期性 Nudge | ✅ 每 10 轮 | ❌ | ❌ |
| Session 结束 | ✅ `on_session_end` | ❌ | ❌ |
| 文件系统监听 | ❌ | ✅ Chokidar | ❌ |
| Session 开始 | ✅ 加载 | ✅ 同步索引 | ❌ |
| 上下文溢出 | ❌ (另有一套压缩) | ❌ | ✅ Compaction |
| 自定义间隔 | ❌ | ✅ intervalMinutes | ❌ |

### 2.3 存什么

| 项目 | 内容粒度 | 不存的内容 |
|------|---------|-----------|
| Hermes | 单条声明式事实（"User prefers concise responses"） | 任务进度、commit hash、"修复了 X bug" |
| OpenClaw | 文件 chunk（原文段落 + embedding 向量） | — |
| OpenCode | 结构化会话摘要（Goal/Progress/Decisions/Next Steps） | — |

**Hermes 的"不存什么"规则非常关键**：
> "Temporal knowledge — task progress, commit SHAs, 'fixed bug X' — should NOT go in memory. Use session_search to retrieve that from historical transcripts."

### 2.4 怎么用

| 项目 | 注入方式 | 注入位置 | 频率 |
|------|---------|---------|------|
| Hermes 内置 | System Prompt（frozen snapshot） | 最顶层 system message | 每个 session 一次 |
| Hermes 外部 | User Message（`<memory-context>` 标签） | 当前 user message 尾部 | 每个 turn |
| OpenClaw | System Prompt（当前记忆上下文） | system message 块 | 每个 session |
| OpenCode | Compaction summary 作为历史消息 | messages 数组 | 每次 compaction 后 |

### 2.5 什么时候用

| 时机 | Hermes | OpenClaw | OpenCode |
|------|--------|----------|----------|
| Session 启动 | ✅ 加载 MEMORY.md | ✅ 同步索引 | ✅ 加载 AGENTS.md |
| 每个 Turn | ✅ Prefetch 外部 Provider | ❌ | ❌ |
| 每次 LLM 调用 | ✅ 注入 cached system prompt | ✅ 系统包含记忆 | ✅ 指令注入 |
| Context 压缩后 | ✅ 重建 system prompt | ❌ | ✅ Compaction summary 替代 |
| Dreaming/Review | ✅ Nudge review | ✅ Dreaming sweep | ❌ |

## 3. NovaClaw 最终方案

### 3.1 三层数据，职责分明

```
MEMORY.md                     Compaction 摘要               Transcript (JSONL)
─────────────────             ──────────────────────         ─────────────────
跨会话持久知识                当前会话进度摘要               完整对话日志
"用户是什么样的人"            "当前做到哪了"                 "谁说了什么"
"项目有什么约定"              "有什么待办"                   "每一步的细节"
§ 分隔的声明式事实            结构化 Markdown                原始消息结构

写入:                        写入:                         写入:
 通道 A: Agent 主动调          上下文溢出时自动               每条消息立即追加
   memory_tool 写入            生成 compaction 消息
 通道 B: 压缩时一并
   提取新事实

召回:                        召回:                         召回:
 memory_search("查询")        [随消息流自动注入]             session_search("查询")
                                                           grep JSONL
```

### 3.2 存储位置

```
{data_dir}/
├── MEMORY.md                          ← 跨会话持久记忆（纯文本）
└── sessions/
    ├── messages/{id}.jsonl            ← 消息记录（含 compaction 消息）
    └── images/{sid}/                  ← 图片实体
```

### 3.3 两层写入

#### 写入通道 A：Agent 主动调用 `memory_tool`（即时）

```
触发: LLM 认为有重要事实需要记住，或用户说"记住..."
动作: memory_tool("add", "User prefers Golang")
      → 追加到 MEMORY.md，以 § 分隔
      → 也可以 replace / remove / list
```

#### 写入通道 B：上下文压缩时自动提取（兜底）

```
触发: 消息数超过 compact_threshold
动作:
  1. 调用 Compaction Agent
  2. 生成结构化摘要 → 存为 JSONL 消息（type: "compaction"）
  3. 摘要中的持久事实 → 同步写入 MEMORY.md
  4. 摘要替代被压缩的旧消息，参与后续对话
```

### 3.4 两个工具：写、搜全在一入口

```
memory (action=add/replace/remove/list/search)
  写入: memory("add", "User prefers Golang")
  搜索: memory("search", "用户喜欢什么语言")
  → 扫描 MEMORY.md → 返回匹配条目

session_search (query, limit)
  搜索: session_search("之前修复的 login bug")
  → 扫描 sessions/{id}.jsonl → 返回匹配消息
```

> 写和搜合一（`memory`），因为 LLM 只需记一个工具名。通过 `action` 参数切换行为：`add`/`replace`/`remove`/`list`/`search`。

### 3.5 完整交互流程

```
用户: "我之前说喜欢什么语言来着？"
  ↓
LLM 判断: 当前消息里没有 → 调用 memory_search("编程语言偏好")
  ↓
memory_search → 扫描 MEMORY.md → "User prefers Golang"
  ↓
LLM: "你之前说过喜欢 Golang"

---

上下文压缩时:
AgentRuntime.compact →
  Compaction Agent 生成摘要:
  ```markdown
  ## Progress
  - 完成 login 模块重构
  ## Decisions
  - 选择 tokio 作为异步运行时
  ```
  → 摘要存为 JSONL compaction 消息
  → 发现"选择了 tokio"是新事实 → 写入 MEMORY.md
```

### 3.6 搜索实现：起步用纯文本

| 阶段 | 检索方式 | 说明 |
|------|---------|------|
| **现在** | 全文扫描 grep | MEMORY.md 几百行，毫秒级，够用 |
| **未来** | SQLite FTS5 | 当 MEMORY.md 到几千行时上，不用改 API |

> **不做向量搜索**: 当前场景是单用户本地，记忆条目可控（默认 800 tokens 限制）。全文匹配已经足够精准，向量搜索增加 embedding 模型依赖，现阶段不值。

## 4. 图片支持集成

### 4.1 存储架构

```
{data_dir}/
├── MEMORY.md                          ← 跨会话持久记忆
└── sessions/
    ├── messages/{id}.jsonl            ← 消息记录（含 compaction 消息）
    └── images/{sid}/                  ← 图片实体文件
        ├── {uuid1}.jpg
        ├── {uuid2}.png
        └── {uuid3}.jpg
```

- JSONL 只存图片路径引用（`image_paths: ["uuid1.jpg"]`），不存 base64
- 图片实体存为独立文件，按 session 分目录
- 发送给 LLM 时临时用 base64 data URL（OpenAI vision 格式）

### 4.2 数据流

```
发送时:
  Ctrl+V 粘贴 / 附件按钮 → FileReader → base64
  → POST /api/chat/stream { message, images: ["data:image/jpeg;base64,..."] }
  → 后端: 解码 base64 → 写 sessions/images/{sid}/{uuid}.jpg
  → 后端: content 组装为 [{type:"text",...},{type:"image_url",...}]
  → 发给 LLM
  → JSONL 记录: { content: "文本", image_paths: ["uuid.jpg"] }

加载历史:
  GET /api/session → 返回 messages (含 image_paths)
  → 前端: 对每条 image_path → GET /files/image/{sid}/{path}
  → <img src="..."> 渲染
```

### 4.3 后端改动

#### 4.3.1 `storage::Message` 新增字段

```rust

pub message_type: Option<String>,      // "compaction" 或 None
```

#### 4.3.2 `llm/types.rs` — content 支持多模态

```rust

// 带图:   content = [{"type":"text","text":"..."},
//                     {"type":"image_url","image_url":{"url":"data:..."}}]

}
```

### 4.4 前端改动

#### 4.4.1 类型定义

```typescript
export interface Message {
  imagePaths?: string[]  // 历史消息图片路径
  images?: string[]      // 流式消息临时 base64（不持久化）
}
```

#### 4.4.2 输入框 — Ctrl+V 粘贴图片

```tsx
const [pendingImages, setPendingImages] = useState<string[]>([])

// 粘贴事件
const handlePaste = (e: ClipboardEvent) => {
  for (const item of e.clipboardData?.items ?? []) {
    if (!item.type.startsWith('image/')) continue
    const file = item.getAsFile()
    if (!file) continue
    const reader = new FileReader()
    reader.onload = () => setPendingImages(prev => [...prev, reader.result as string])
    reader.readAsDataURL(file)
  }
}

// 拖拽事件（同逻辑）
```

#### 4.4.3 附件按钮 — 点击上传图片

在输入框旁增加 📎 按钮，点击触发 `<input type="file" accept="image/*" multiple>`。

#### 4.4.4 发送时携带图片

```tsx
startChatStream({
  message: userText,
  images: pendingImages,  // ["data:image/jpeg;base64,..."]
})
setPendingImages([])
```

#### 4.4.5 图片缩略图预览（发送前）

```tsx
{pendingImages.map((url, i) => (
  <div key={i} className="relative">
    <img src={url} className="w-16 h-16 rounded object-cover" />
    <button onClick={() => removeImage(i)}>✕</button>  {/* 移除 */}
  </div>
))}
```

#### 4.4.6 历史消息渲染图片

```tsx
{msg.imagePaths?.map((path) => (
  <img
    src={`/api/files/image/${msg.sessionId}/${path}`}
    className="max-w-[200px] rounded"
    onError={(e) => e.currentTarget.alt = '图片已失效'}
  />
))}
```

### 4.5 图片压缩策略

参考 OpenClaw：前端发送前压缩，避免传输大文件。

```typescript
// lib/imageCompress.ts
export async function compressImage(file: File): Promise<string> {
  // 1. 解码为 ImageBitmap
  // 2. 缩放: max(width, height) ≤ 1024px
  // 3. Canvas 绘制 + JPEG 输出 (quality 0.8)
  // 4. 返回 data:image/jpeg;base64,...
}
```

| 场景 | 处理 |
|------|------|
| 小图（<500KB） | 直接 base64，不压缩 |
| 大图（>1MB） | 缩放 ≤1024px + JPEG q=0.8 |
| 非 JPEG/PNG | 统一转 JPEG |

## 6. 记忆提示词注入设计

提示词是记忆系统的"说明书"——LLM 通过它知道记忆工具的存在、用法和时机。

### 6.1 System Prompt 注入 MEMORY.md（Session 启动时）

```
注入时机: Session 创建时（首次构建 system prompt）
注入位置: SystemPromptBuilder 的 "Context" 段
注入内容: MEMORY.md 的 frozen snapshot（只读，本 session 不变）
容量控制: 截断到 ~800 tokens，超出部分加 "…(truncated)" 标记
```

**示例**：

```text
## Persistent Memory

The following facts are remembered from previous sessions:

§ User prefers concise responses with bullet points
§ User runs Windows 11, uses VS Code as primary editor
§ Project convention: snake_case for Rust, camelCase for TypeScript
§ API keys should never be hardcoded — use .env files

Use these facts to personalize your responses. If you learn new facts 
in this session, use the `memory_tool` to save them.
```

### 6.2 `memory` 工具描述

LLM 通过工具 schema 的 `description` 字段理解工具的用途：

```
工具名: memory
描述:
  Manage persistent cross-session memory and search past knowledge.
  Write facts to MEMORY.md so they persist across conversations;
  search MEMORY.md to recall what was previously remembered.

参数:
  action: "add" | "replace" | "remove" | "list" | "search"
  content: string  — add/replace 时的事实文本
  query: string    — search 时的搜索关键词
  category: string — 可选分类标记（如 "preference", "convention", "environment"）

操作指南:
  add:
    - 用户说"记住…"时立即调用
    - 发现重要的项目约定或偏好时主动存入
    - 用宣言式语言: "User prefers Golang"
  replace:
    - 用户纠正错误时更新已有事实（提供旧文本 + 新文本）
  remove:
    - 删除不再适用的事实
  search:
    - 用户问"我之前说过…吗？"
    - 当前对话不包含用户提到的历史信息
    - 需要确认之前的项目约定或决策
  list:
    - 查看所有已存记忆

不存什么:
  - 任务进度、commit SHA、"修复了 X bug"（这些用 session_search）
  - 会过期的临时信息
  - 每条记忆简洁独立，不超过一句话
```

### 6.3 `session_search` 工具描述

```
工具名: session_search
描述:
  Search the current session's transcript (JSONL) for temporal
  information. Use this instead of memory/search for session-specific
  or temporary info like task progress, bug details, past decisions.

参数:
  query: string — 搜索关键词
  limit: number — 返回条数上限（默认 5）

使用时机:
  - 用户问"刚才那个 bug 是什么来着？"
  - 需要回顾当前会话的具体技术细节
  - 已被上下文压缩的历史消息
```

### 6.5 Compaction Agent 提示词

压缩时使用的专用 prompt，引导 LLM 生成结构化摘要：

```text
[Agent 已达到上下文限制。请总结以下对话内容，生成结构化摘要：]

## Goal
- [这个会话的目标，一句话总结]

## Progress
### Done
- [已完成的工作项]
### In Progress
- [正在进行的工作]
### Blocked
- [阻塞项，如有]

## Decisions Made
- [关键决策及理由]

## Critical Context
- [重要技术细节、错误信息、开放问题]

## Relevant Files
- [文件路径: 修改原因]

此外，从上述对话中提取需要**跨会话记住的持久事实**：
- 用户偏好（语言、工具、回复风格）
- 项目约定（命名规范、部署方式）
- 环境信息（操作系统、已安装的工具）

这些事实将通过 memory_tool 自动写入 MEMORY.md。
```

### 6.6 注入时机总结

```
Session 启动:
  └→ System prompt 注入 MEMORY.md snapshot（frozen，~800 tokens）

每次 LLM 调用:
  └→ 工具列表中包含 memory / session_search

上下文压缩时:
  └→ Compaction prompt 注入 → LLM 生成摘要 + 提取持久事实

压缩后重启:
  └→ System prompt 重建 → 重新加载 MEMORY.md（此时包含新事实）
```
