# NovaClaw 消息存储与图片存储方案

## 1. 业界参考分析

先看三个成熟项目的做法：

### 1.1 Hermes Agent（Python / Claude 官方）

| 维度 | 做法 |
|------|------|
| 主存储 | **SQLite**（WAL 模式），`~/.hermes/state.db` |
| 消息表结构 | `messages(id, session_id, role, content TEXT, tool_calls TEXT, ...)` |
| 图片消息格式 | content 为 `[{"type":"text","text":"..."},{"type":"image_url","image_url":{"url":"data:image/png;base64,...."}}]` |
| content 列编码 | 纯文本直接存；多模态列表 JSON 序列化后加 `\x00json:` 前缀存入 TEXT 列 |
| **图片是否入库** | **否** — 写入 SQLite 前剥离 base64，替换为 `[screenshot]` 占位符 |
| 冗余备份 | `sessions/session_{id}.json`（每个会话一个 JSON 文件，含完整消息数组） |
| 全文搜索 | FTS5 + trigram（支持 CJK/中文） |
| 并发 | SQLite WAL 多读单写 |
| Schema 管理 | `schema_version` 表，当前版本 11 |

**核心结论**：图片仅在内存中流转并发送给 LLM，不持久化。历史回放时图片消失，显示 `[screenshot]`。

### 1.2 OpenClaw（Node.js/TypeScript，JSONL 为主）

| 维度 | 做法 |
|------|------|
| 主存储 | **JSONL 文件**，`sessions/<id>.jsonl`（一行一条 JSON 记录） |
| 消息格式 | `{"type":"message","id":"uuid","timestamp":"ISO","message":{role, content, model, usage, cost, ...}}` |
| 会话元数据 | `sessions/sessions.json`（单文件，原子写 + 排他锁） |
| **图片存储** | **Base64 内嵌**于 JSONL 消息的 `content[].data` 字段 |
| 图片净化 | 加载时自动 resize（最大 1200px）+ recompress（最大 5MB）+ 统一 JPEG |
| SQLite 用途 | **仅用于 Memory 向量搜索**（`node:sqlite` + `sqlite-vec` + FTS5），不存消息 |
| 并发控制 | 文件级写锁 + 队列化 append（`transcriptAppendQueues`） |
| 归档 | 重命名为 `.jsonl.<reason>.<timestamp>` |

**核心结论**：JSONL 存 base64 图片，但有严格的大小限制（1200px / 5MB）。同一 JSONL 文件同时包含文本和图片，简单但历史加载重。

### 1.3 OpenCode（TypeScript + SQLite + Event Sourcing）

| 维度 | 做法 |
|------|------|
| 主存储 | **SQLite**（WAL，64MB cache），事件溯源模式 |
| 消息表 | `message(id, session_id, data JSON)` — 单 JSON 列存完整消息 |
| Parts 表 | `part(id, message_id, session_id, data JSON)` — 12 种 Part 类型 |
| **图片存储** | **Base64 内嵌**于 Part 的 `data.url` 字段（通过 `data:image/png;base64,...`） |
| 图片来源 | 用户粘贴/拖拽、Read Tool 读文件、MCP 工具返回 |
| MIME 嗅探 | 魔数检测（PNG/JPEG/GIF/Webp/PDF） |
| 流式更新 | `INSERT ... ON CONFLICT DO UPDATE`（upsert） |
| 并发 | SQLite WAL + 事务锁 |
| 双运行时 | Bun（`bun:sqlite`）/ Node.js（`node:sqlite`） |
| JSONL 用途 | **仅前端本地缓存**（prompt-stash、frecency），不参与消息持久化 |
| 事件溯源 | `SyncEvent.run()` → 单事务写入 event + projector → 业务表 |

**核心结论**：图片 base64 直接存在 SQLite 的 JSON 字段里。因为 SQLite 处理大 document 没问题，且 JSON 列不需要为图片单独建表。事件溯源保证一致性。

### 1.4 对比总结

| | Hermes | OpenClaw | OpenCode |
|---|---|---|---|
| 存储引擎 | SQLite | JSONL | SQLite |
| 图片入库 | ❌ 剥离为占位符 | ✅ base64 内嵌 | ✅ base64 内嵌 |
| 图片净化 | 被动重试缩小 | 主动 resize+recompress | 依赖来源端控制 |
| 搜索 | FTS5 | SQLite-vec | FTS5 |
| 消息格式 | TEXT 列 + \x00json: 前缀 | 一行 JSON | JSON 列 |
| 并发 | WAL | 文件锁 | WAL + 事务 |

## 2. NovaClaw 现状分析

### 2.1 当前架构

```
存储层：JSONL 文件（sessions/<id>.jsonl）
消息结构：storage::Message (Rust struct → serde_json → JSONL)
图片支持：无（ChatMessage.content 为纯 String）
运行环境：Rust (axum) + React (TypeScript)
并发模式：单用户本地（无需分布式）
```

### 2.2 约束条件

- Rust 后端，不能直接用 `better-sqlite3` 等 Node.js 库
- 当前 JSONL 追加写入已经在用，改动最小优先
- 多模态（图片）是新需求，不是重构消息存储的理由
- 历史消息加载时图片需能正常显示

### 2.3 目标

1. 支持前端传入图片 → 发给 LLM vision 模型
2. 图片随消息持久化，历史回放时可显示
3. 不显著增加 JSONL 文件体积（避免单行 4MB）
4. 改动范围可控

## 3. 推荐方案

### 3.1 核心决策：**图片外存 + JSONL 引用**

结合三个项目的经验，采用以下设计：

```
sessions/messages/{session_id}.jsonl     ← 消息记录（轻量，只存图片路径引用）
sessions/images/{session_id}/
    ├── {uuid1}.jpg                      ← 图片实体文件
    └── {uuid2}.jpg
```

**为什么**：
- Hermes 证明了"图片不落 SQLite"是对的——持久化没必要存大 base64
- OpenClaw 内嵌 base64 到 JSONL 会导致历史加载卡顿
- OpenCode 存 SQLite JSON 列没问题，但 NovaClaw 目前没有 SQLite
- **文件外存 + 路径引用**：JSONL 保持轻量，图片按需加载，最契合当前架构

### 3.2 数据流

```
┌── 发送 ──────────────────────────────────────────────────────────────┐
│                                                                       │
│  前端拖拽/粘贴图片                                                     │
│    ↓ FileReader → base64                                              │
│    ↓ (可选) 前端压缩：resize ≤1024px, JPEG q=0.8                      │
│                                                                       │
│  POST /api/chat/stream                                                │
│  {                                                                    │
│    "message": "这张图是什么？",                                        │
│    "images": [                                                        │
│      "data:image/jpeg;base64,/9j/4AAQ..."                            │
│    ]                                                                  │
│  }                                                                    │
│    ↓                                                                  │
│  后端 chat.rs:                                                        │
│    1. 解码 base64 → 写 sessions/images/{sid}/{uuid}.jpg               │
│    2. 构建 content: [                                                 │
│         {"type":"text","text":"这张图是什么？"},                       │
│         {"type":"image_url","image_url":{"url":"data:image/..."}}     │
│       ]                                                               │
│    3. 发给 LLM                                                        │
│    4. 写到 jsonl 的是 content 文本 + image_paths: ["uuid.jpg"]        │
│                                                                       │
└───────────────────────────────────────────────────────────────────────┘

┌── 加载历史 ───────────────────────────────────────────────────────────┐
│                                                                       │
│  GET /api/session?session_id=xxx                                      │
│    ↓ 读取 jsonl，解析每条消息                                         │
│    ↓ user 消息附带 image_paths: ["abc123.jpg", "def456.jpg"]          │
│                                                                       │
│  前端渲染:                                                             │
│    对每条 image_path → GET /files/image/sessions/images/{sid}/{path}  │
│    → blob → <img src={blobUrl}>                                       │
│    （或直接 <img src="/api/files/image/sessions/images/..."> ）       │
│                                                                       │
└───────────────────────────────────────────────────────────────────────┘
```

### 3.3 为什么不用 SQLite

| 维度 | 文件外存 | SQLite BLOB |
|------|----------|-------------|
| 图片 HTTP 服务 | 直接用静态文件服务，不占后端内存 | 需要额外 API 从 BLOB 转 stream |
| 图片清理 | `rm -rf sessions/images/<sid>/` | DELETE + VACUUM |
| 备份迁移 | `cp -r sessions/` | 需要 SQL dump |
| 引入成本 | 3-4 个文件改动 | 消息层全量重写 |
| 适合 NovaClaw? | ✅ 现在就能做 | ❌ 大重构，等需要搜索时再说 |

## 4. 详细设计

### 4.1 后端改动

#### 4.1.1 `storage.rs` — Message 结构体

```rust
pub struct Message {
    // ... 现有字段保持不变 ...
    pub id: String,
    pub session_id: String,
    pub role: String,
    pub content: String,
    // ... 省略已有字段 ...

    /// 图片引用路径列表（相对于 sessions/images/{session_id}/）
    #[serde(default, skip_serializing_if = "Option::is_none")]
    pub image_paths: Option<Vec<String>>,
}
```

> `content` 字段保持纯文本（发给前端显示用），图片路径单独字段，方便前端按需加载。

#### 4.1.2 `llm/types.rs` — ChatMessage.content 类型

```rust
// content 从 String 改为 serde_json::Value
// 纯文本消息:  content = "hello"
// 带图消息:    content = [{"type":"text","text":"..."},
//                         {"type":"image_url","image_url":{"url":"data:..."}}]
pub struct ChatMessage {
    pub role: String,
    pub content: serde_json::Value,  // ← 改类型
    // ...
}
```

序列化逻辑：
```rust
impl ChatMessage {
    fn content_value(text: &str, image_urls: &[String]) -> serde_json::Value {
        if image_urls.is_empty() {
            serde_json::Value::String(text.to_string())
        } else {
            let mut parts: Vec<serde_json::Value> = vec![serde_json::json!({
                "type": "text", "text": text
            })];
            for url in image_urls {
                parts.push(serde_json::json!({
                    "type": "image_url",
                    "image_url": { "url": url }
                }));
            }
            serde_json::Value::Array(parts)
        }
    }
}
```

#### 4.1.3 `server/routes/chat.rs` — 接收图片

```rust
#[derive(Deserialize)]
struct ChatStreamReq {
    session_id: Option<String>,
    message: String,
    model: Option<String>,
    workspace: Option<String>,
    #[serde(default)]
    images: Vec<ImageAttachment>,  // ← 新增
}

#[derive(Deserialize)]
struct ImageAttachment {
    /// data:image/png;base64,iVBORw...
    url: String,
}
```

处理逻辑：
```rust
// 1. 解码 base64，存到文件
let image_dir = get_images_dir().join(&session_id);
fs::create_dir_all(&image_dir)?;
let mut image_paths = Vec::new();
for img in &req.images {
    let (mime, data) = parse_data_url(&img.url)?;  // 提取 mime, 解码
    let ext = mime_to_ext(&mime);                   // image/png → png
    let filename = format!("{}.{}", Uuid::new_v4(), ext);
    let filepath = image_dir.join(&filename);
    fs::write(&filepath, &data)?;
    image_paths.push(filename);
}

// 2. 构建 ChatMessage（content 为 vision 格式数组）
let content = ChatMessage::content_value(&req.message, &image_urls);

// 3. 存 jsonl 时记录 image_paths
let storage_msg = storage::Message {
    // ...
    content: req.message.clone(),     // 存纯文本
    image_paths: Some(image_paths),   // 存图片路径
};
```

#### 4.1.4 `server/routes/files.rs` — 图片读取 API

```rust
// GET /files/image/:session_id/:filename
async fn get_image(Path((session_id, filename)): Path<(String, String)>) 
    -> impl IntoResponse 
{
    let path = get_images_dir().join(&session_id).join(&filename);
    if !path.exists() {
        return StatusCode::NOT_FOUND.into_response();
    }
    let bytes = std::fs::read(&path)?;
    let mime = mime_guess::from_path(&filename)
        .first_or_octet_stream();
    ([(header::CONTENT_TYPE, mime.as_ref())], bytes).into_response()
}

fn get_images_dir() -> PathBuf {
    crate::config::get_sessions_dir().join("images")
}
```

#### 4.2.2 输入框支持图片 — `ChatPanel.tsx`

```tsx
const [pendingImages, setPendingImages] = useState<string[]>([])

// 粘贴事件
const handlePaste = (e: ClipboardEvent) => {
  const items = e.clipboardData?.items
  if (!items) return
  for (const item of items) {
    if (item.type.startsWith('image/')) {
      const file = item.getAsFile()
      if (!file) continue
      const reader = new FileReader()
      reader.onload = () => setPendingImages(prev => [...prev, reader.result as string])
      reader.readAsDataURL(file)
    }
  }
}

// 拖拽事件
const handleDrop = (e: DragEvent) => {
  e.preventDefault()
  for (const file of e.dataTransfer?.files ?? []) {
    if (!file.type.startsWith('image/')) continue
    const reader = new FileReader()
    reader.onload = () => setPendingImages(prev => [...prev, reader.result as string])
    reader.readAsDataURL(file)
  }
}
```

发送时：
```tsx
startChatStream({
  message: userContent,
  session_id: sessionId,
  workspace: workspacePath,
  images: pendingImages,  // ← 传递图片
}, callbacks)
setPendingImages([])  // 清空
```

## 5. 边界情况

| 场景 | 处理 |
|------|------|
| 图片超大（>5MB） | 前端压缩：resize ≤1024px宽，JPEG quality 0.8 |
| 图片格式不支持 | 后端 MIME sniff → 不支持的返回错误 |
| 历史消息无图片 | image_paths 字段为 None/空，前端不渲染图片区 |
| 图片文件被误删 | 前端 `<img onerror>` → 显示"图片已失效"占位符 |
| 同一会话多次传图 | 每次生成新 UUID 文件名，不覆盖 |
| 清理会话 | 删除 jsonl + `rm -rf sessions/images/{sid}/` |
| 纯文本消息 | content 仍是 `String`（非数组），不触碰图片逻辑 |

## 6. 实现顺序

| Phase | 内容 | 涉及文件 |
|---|---|---|
| **P1** | ChatMessage.content → serde_json::Value | `llm/types.rs`, `agent/runtime.rs` |
| **P2** | 图片接收 + 磁盘存储 + jsonl 引用 | `server/routes/chat.rs`, `storage.rs` |
| **P3** | 图片读取 API | `server/routes/files.rs` |
| **P4** | 前端图片输入（粘贴/拖拽） | `ChatPanel.tsx`, `useApi.ts`, `types/index.ts` |
| **P5** | 前端历史图片渲染 | `ChatMessages.tsx` |
| **P6** | 前端图片压缩（可选优化） | 新建 `lib/imageCompress.ts` |

总计约 6-8 小时。

## 7. 未来演进：SQLite 迁移

当前方案充分满足需求。当以下条件之一触发时，考虑整体迁移到 SQLite：

1. **消息全文搜索** — 需要 FTS5 索引
2. **消息分页/滚动加载** — JSONL 只能全文扫描，SQL 可分页
3. **多实体关联查询** — session → message → image 的 JOIN
4. **用户需求明确** — 如"按月查询历史对话"等

迁移路径：
```
1. 建 SQLite schema（message 表 + image_blob 表）
2. 双写一段时间（同时写 jsonl + sqlite）
3. 读切换到 sqlite
4. 删 jsonl 写 + 旧 jsonl 数据导入
```
